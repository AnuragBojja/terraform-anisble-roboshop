# Redis Role Documentation

## Component Purpose
Redis provides in-memory data storage used by `cart` and `user` services for fast access patterns.

## Source Artifacts Used
- `redis.yaml`

## Inputs and Dependencies
- Inventory group: `redis`
- OS package manager with module streams
- Service name: `redis`

## Task-by-Task Breakdown
1. Disable default Redis module
What: runs `dnf module disable redis -y`.
Why: avoids installing an older default stream.

2. Enable Redis 7 module
What: runs `dnf module enable redis:7 -y`.
Why: pins deployment to Redis 7 stream expected by app behavior.

3. Install Redis package
What: installs `redis`.
Why: installs server binary and service unit.

4. Allow remote connections
What: replaces `127.0.0.1` with `0.0.0.0` in `/etc/redis/redis.conf`.
Why: permits app nodes to connect over network.

5. Disable protected mode
What: sets `protected-mode no` in config.
Why: allows external connections in non-local deployment topology.

6. Start and enable Redis
What: starts Redis and enables boot start.
Why: ensures runtime availability and persistence across reboot.

## Configuration Details
- Network settings are intentionally opened for multi-host architecture.
- No password/TLS configuration is added by this role.

## Service Lifecycle
- Final expected state: service started and enabled.

## Data Bootstrap
- Not applicable. Redis data is generated by runtime app behavior.

## Validation and Expected Results
- `systemctl status redis` should be active/running.
- `ss -lntp | grep 6379` should show listening socket.
- Remote check from app node:
  - `redis-cli -h redis.anuragaws.shop ping`
  - Expected output: `PONG`.

## Risks and Operational Notes
- `protected-mode no` + bind `0.0.0.0` is insecure without network controls.
- Uses `command` module for module operations; not fully state-aware.
